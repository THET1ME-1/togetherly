# Переезд на Supabase — план публичного релиза (безопасно)

Цель: выложить обновление в общий доступ и начать переезд как можно раньше,
**без утечки данных и без потери сообщений** при поэтапном обновлении пар.

Переезд разбит на 3 стадии. Публиковать можно уже после Стадии 2 — это и есть
«как можно раньше». Стадия 3 (экономия на чтениях) — позже, после теста на телефоне.

---

## Что уже сделано в ветке `feature/supabase-migration`

- **Детектор смешанных пар включён** — приложение определяет, оба ли партнёра
  на новой сборке (маркер `sbMig` в group-doc). Поведение не меняет, просто
  собирает инфу. (`firebase_service.dart`)
- **Личность для Supabase = Firebase-токен** — `Supabase.initialize(accessToken:)`
  в `lib/main.dart` шлёт Firebase ID-токен в каждый запрос.
- **Миграция защиты `supabase/security.sql`** — включает RLS на всех таблицах,
  привязывает доступ к Firebase UID.
- **`enabledForEveryone = false`** — публичные юзеры пока полностью на Firebase
  (нулевой риск). Включим в Стадии 2.

---

## СТАДИЯ 1 — Замок на склад (безопасность). Делать ПЕРВОЙ.

Порядок строгий — иначе тест-аккаунты временно перестанут видеть данные.

1. **Supabase Dashboard → Authentication → Sign In / Providers → Third-Party Auth
   → Add provider → Firebase.** Указать Firebase Project ID
   (он же в `google-services.json` → `project_id`). Сохранить.
   _Это учит Supabase доверять входу через Firebase._

2. **Собрать и запустить приложение с этой ветки** (токен уже в `main.dart`).
   Зайти тест-аккаунтом. Проверить, что Supabase-данные грузятся как раньше
   (RLS пока ВЫКЛЮЧЕН). _Это подтверждает, что токен принимается._
   - Если данные пропали → провайдер Firebase в п.1 настроен неверно. НЕ идти дальше.

3. **Только теперь** выполнить `supabase/security.sql`
   (Dashboard → SQL Editor → вставить файл → Run).

4. **Проверить с двумя тест-аккаунтами:**
   - [ ] Свои чаты / память / настроения / виджет — видны и редактируются.
   - [ ] Пара видит друг друга (общая группа работает).
   - [ ] Приглашение/паринг работает (если идёт через Supabase — см. «Известные сценарии»).
   - [ ] С третьего аккаунта (не в паре) чужая группа не видна.

5. **Перезапустить функции с защитой** — после `security.sql` выполнить заново
   `supabase/coins.sql` и `supabase/mascots.sql` (в них добавлена проверка
   личности/членства). Порядок важен: сначала `security.sql` (там хелперы
   `app_require_uid` / `is_group_member`), потом эти два файла.
   - Прочие RPC (`group_set_member_*`, `memory_patch`, `canvas_*`,
     `increment_miss_you`) — `LANGUAGE sql` без `SECURITY DEFINER`, выполняются
     с правами вызывающего → уже под RLS, доп. защита не нужна.
   - **Публиковать можно только после того, как и этот шаг применён.**

6. **ФИНАЛЬНЫЙ ШАГ — перезапустить `security.sql` ПОСЛЕДНИМ.** `security.sql`
   идемпотентен; гоняем его в самом конце, чтобы гарантированно включить RLS на
   всех таблицах, даже если какой-то `*.sql` запускался после него.
   - _Был баг (исправлен):_ файлы создания таблиц (`schema.sql`, `coins.sql`,
     `mascots.sql`, `canvas.sql`, …) раньше заканчивались
     `ALTER TABLE … DISABLE ROW LEVEL SECURITY` (рудимент Фазы 1). Поэтому
     повторный прогон `coins.sql`/`mascots.sql` из шага 5 СНИМАЛ RLS с
     `iap_purchases` и `mascots` → они светились «UNRESTRICTED». Строки `DISABLE`
     удалены из всех файлов; теперь RLS управляет только `security.sql`, и
     повторный прогон любого файла защиту не снимает.
   - **Если таблицы `iap_purchases` / `mascots` показаны UNRESTRICTED — просто
     запусти `security.sql` ещё раз; он включит RLS и навесит политики.**

## СТАДИЯ 2 — Публичный релиз с двойной записью (переезд начинается)

Когда Стадия 1 проверена:

- Включить запись в Supabase для всех, но **чтение оставить из Firebase**:
  пара всегда видит друг друга (Firebase — общий источник), старые версии
  работают, ничего не теряется, а данные начинают копиться в Supabase.
- Это безопасно для смешанных пар по построению (нет переключения источника).
- _Код этой стадии — следующий после Стадии 1; нужен сплит гейта на «запись»
  (всем) и «чтение» (пока Firebase)._

После этого можно публиковать обновление в общий доступ.

---

## СТАДИЯ 3 — Переключение чтения на Supabase (экономия денег)

- Группа переключается на чтение из Supabase **только когда детектор подтвердил,
  что оба партнёра на новой сборке** (`migGroupFull`). Переключение — на следующей
  сессии, без потери данных.
- **Требует проверки на реальном телефоне** — ошибка в порядке «резолв → бинд
  листенеров» = потеря сообщений. Без прогона приложения не включать.

---

## Известные сценарии под проверку (RLS строгий — это ожидаемо)

- **Паринг по invite-code**: ищет чужого юзера, который ещё НЕ в общей группе →
  под RLS `users` это запрещено. Если паринг идёт через Supabase — нужен
  отдельный SECURITY DEFINER RPC поиска по коду. Если через Firebase — ок.
- **Realtime**: стримы (группа/чат/настроения) под RLS отдают только свои строки.
  Проверить, что live-обновления приходят залогиненному юзеру.
- **Storage**: media приватный (Signed URL), avatars — публичное чтение, запись —
  только авторизованным. Привязку к членству группы по пути файла можно добавить позже.

## Две дыры, которые закрываем (из аудита)

1. ✅ **Таблицы открыты** — закрыто `security.sql` (RLS + Firebase-токен).
2. ✅ **Денежные/DEFINER-функции** (`purchase_*`, `grant_*`, `group_record_activity`)
   принимали `uid`/`group_id` параметром и обходили RLS → любой мог начислить себе
   монеты или влезть в чужую группу. Закрыто: в `coins.sql` добавлен
   `app_require_uid(p_uid)` (11 функций), в `mascots.sql` — проверка членства.
   Нужно ПЕРЕЗАПУСТИТЬ эти файлы после `security.sql` (см. Стадия 1, шаг 5).
