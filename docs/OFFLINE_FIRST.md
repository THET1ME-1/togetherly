# Offline-First (sembast) — план и статус

## Зачем
Сейчас приложение читает данные напрямую с сервера PocketBase по сети: открыл
экран → запрос в интернет → данные. **Нет сети — пустой экран.** Каждое
переоткрытие экрана повторно тянет весь список. Цель — архитектура **Offline
First**: UI всегда читает из локальной копии на устройстве (мгновенно, офлайн),
а сервер только досыпает изменения в фоне и принимает наши правки.

## Ключевые решения
- **Локальная БД:** `sembast` (чистый Dart, NoSQL-документы). Без нативного кода
  → сборки iOS/RuStore/Codemagic не затрагиваются. PB-записи — JSON, ложатся 1:1.
- **Объём:** всё приложение (чтения + запись + офлайн-медиа).
- **Синхронизация — инкрементальная:** пагинация + дельта по `updated`. Уже
  загруженное **повторно не качаем**. (Это сознательно отменяет прежнюю директиву
  «грузить весь список без лимитов».)
- **Медиа-инфра:** пока как есть (PB protected files + токен). Клиентский слой
  делаем агностично к источнику (кэш байтов по стабильному ключу `pb://`), чтобы
  переезд на **S3 + CDN с подписанными ссылками** был drop-in позже. Публичный
  CDN нельзя — фото пар приватные.

## Архитектура: один узел чтения, один узел записи
- **Чтения** проходят через ДВА генерик-метода: `PbRealtimeService.watchList` и
  `watchRecord` (`lib/services/pb_realtime_service.dart`). Делаем их
  кэш-ориентированными → офлайн получает всё приложение почти даром.
- **Записи** идут через `PbDataService` (`lib/services/pb_data_service.dart`),
  обёрнутый репозиториями. Рядом ставим **очередь отправки** (outbox).
- PB принимает **клиентский id при create** → офлайн-созданная запись сразу
  получает финальный id, ремаппинг не нужен.

### Поток чтения (offline-first) — Фаза 1
1. `watchList` отдаёт поток из локального кэша (sembast, реактивно) — мгновенно,
   офлайн.
2. Появилась сеть → **дельта-синхронизация**: тянем только `updated > водяной_знак`
   (страницами), upsert в кэш, двигаем знак. Полная сверка — редко (pull-to-refresh
   / маленькие коллекции).
3. SSE-подписка пишет дельты (create/update/delete) сразу в кэш.
4. UI всегда привязан к кэшу → отражает оптимистичные правки, дельты и сверку.
   Сохраняем сортировку `compare`, RT-3 backoff, защиту гонки onCancel.

### Поток записи (outbox) — Фазы 2–3
- Мутации репозитория: (1) оптимистично пишем в кэш → UI обновляется сразу;
  (2) кладём операцию в очередь; (3) если онлайн — сразу шлём.
- Типы операций: **verbatim** (create/update/delete по id/фильтру — идемпотентны);
  **semantic RMW** (toggleSaved/инкременты/реакции — переигрываем по свежему
  состоянию сервера, не «слепой» перезаписью); **server-atomic** (group-роуты).
- Порядок FIFO; коалесинг (суммируем инкременты, схлопываем create+delete);
  ретрай с backoff; «ядовитые» операции → отдельный стор + Sentry, очередь не
  блокируют. Live-фичи (presence/typing/курсоры/co-watch/геопозиция) офлайн —
  **no-op**, не ставим в очередь.

### Медиа — Фаза 4
- **Чтение:** кэш байтов по стабильному ключу `pb://` (path_provider); офлайн —
  отдаём локальный файл. Хук в `lib/widgets/storage_image.dart` и плеерах
  `lib/screens/memory_lane/players.dart` (там `pb://`-музыка сейчас вообще не
  резолвится токеном — починим заодно).
- **Запись:** офлайн-созданное медиа ссылается на `localfile://`, показывается
  сразу; при сети грузим через `PbMediaService` → патчим запись `pb://` в кэше и
  на сервере.

## Инвалидация кэша (критично)
Очищать кэш+очередь+медиа при КАЖДОМ выходе/смене пользователя:
`UserData.logout()`, ветка несовпадения uid в `ConnectionsManager._loadLocal`,
`signOut`-вызовы, первый запуск (`prefs.clear()` в `main.dart`). Иначе новый
юзер увидит данные/правки предыдущего.

## Файлы
**Новые (`lib/services/offline/`):** `local_store.dart` (sembast),
`connectivity_service.dart`, `record_scope.dart`, `pb_id.dart`, `backoff.dart`;
далее `outbox_service.dart`, `outbox_op.dart`, `media_cache.dart`.
**Правим:** `pb_realtime_service.dart` (ядро чтений + scope в обёртках),
репозитории (`memory_repository.dart` первым, затем остальные),
`storage_image.dart` + плееры, `main.dart` (init + инвалидация).

## Фазы и статус
- [x] **Фаза 0 — фундамент:** sembast; `LocalStore`, `ConnectivityService`,
  `RecordScope`, `pb_id`, `backoff`; init в `main.dart`.
- [x] **Фаза 1 — офлайн-чтения:** `watchList`/`watchRecord` кэш-ориентированы +
  инкрементальная дельта-синхронизация (водяной знак `updated`, постранично) +
  `scope` на durable-обёртках (воспоминания, комментарии, группа, настроения,
  чат, маскоты, виджет-данные, прочтения, «скучаю», список групп). Live-only
  (typing/presence/курсоры/co-watch/геолокация) и холст — на прежнем пути.
- [x] **Фаза 2a — офлайн-запись (Воспоминания):** очередь `OutboxService` +
  оптимистичный кэш; create/edit/pin/saved/delete воспоминаний работают офлайн и
  досылаются при сети. Инвалидация кэша+очереди при logout/смене юзера
  (`resetOfflineState`, `ensureOwner`). Юнит-тесты ядра кэша (9, зелёные).
- [~] **Фаза 2b — офлайн-запись (почти вся):** ✅ настроения, комментарии, чат,
  **поля группы** (своё настроение/самочувствие/статус) и **состояние маскота**
  (активный/позиция) — оптимистичный кэш + очередь. ⏳ осталось: галерея маскотов
  (create/delete — composite-key id), холст.
- [~] **Фаза 3 — сложная запись (почти вся):** ✅ semantic-RMW полей группы
  (set/clear настроения/статуса — идемпотентны, онлайн через atomic-роуты) +
  счётчики/реакции + **коалесинг очереди** (склейка/аннигиляция повторов с

- [~] **Фаза 4 — медиа (частично):** ✅ отложенная загрузка медиа ВОСПОМИНАНИЙ

## Проверка
Сценарии «в самолёте»: холодный старт офлайн (лента/настроения/чат/маскоты из
кэша, без вечных спиннеров); офлайн создание/правка/удаление → оптимистично →
рестарт офлайн (переживает) → сеть → очередь сливается, партнёр видит по SSE;
сверка при реконнекте (пропущенные удаления/добавления подтягиваются); смена
пользователя (нет чужих данных/правок). Юнит-тесты на `sembast_memory`.
